## 📋 RESUMEN EJECUTIVO

Este plan detalla la implementación de un sistema completo de pagos a cuenta para presupuestos aprobados en MDR, que incluye:

1. **Parte de trabajo para técnicos**: Documento logístico sin precios ni información comercial
2. **Factura Proforma**: Documento idéntico al presupuesto pero con numeración propia para pagos a cuenta del 100%
3. **Factura de Anticipo**: Factura normal por el importe del anticipo con referencia al presupuesto
4. **Factura Rectificativa/Abono**: Anulación de facturas normales con motivo, numeración propia e importes en negativo
5. **Sistema de tracking de pagos**: Nueva tabla para registrar todos los pagos recibidos

**Decisiones clave**:
- ✅ Se registrarán todos los documentos en BD (no solo PDFs)
- ✅ Nueva tabla `pago_presupuesto` para tracking de anticipos
- ✅ Facturas proforma tendrán serie independiente (`FP2024/0001`)
- ✅ **Abonos solo para facturas normales** (NO para facturas proforma)
- ✅ Serie de abonos (`R2024/0001`) ya existe en tabla empresa
- ✅ **Proforma válida también en anticipos**: Al registrar un anticipo, el cliente puede elegir entre recibir una **factura proforma** o una **factura real de anticipo** — ambos tipos tienen el mismo tratamiento funcional
- ✅ **Proforma de anticipo no se abona**: Si la factura emitida por un anticipo es de tipo proforma, en caso de cancelación/error **no se genera abono** — simplemente se vuelve a emitir (se repite)
- ✅ Parte de trabajo ocultará: precios, descuentos, totales y formas de pago
- ✅ **Empresas ficticias vs reales**: Los presupuestos se hacen con empresa ficticia principal, pero **TODAS las facturas (proforma, anticipo, normal) SIEMPRE requieren empresa REAL** para validez fiscal
- 🚫 **Restricción empresa**: Una vez emitida CUALQUIER factura (proforma, anticipo o normal), **NO se permite cambiar de empresa** en ese presupuesto

### 1.4 Actualizar Stored Procedure `sp_obtener_siguiente_numero`

**Archivo**: `BD/migrations/20260216_update_sp_obtener_siguiente_numero.sql`

Añadir soporte para facturas proforma (el tipo 'abono' ya existe):

```sql
DROP PROCEDURE IF EXISTS sp_obtener_siguiente_numero$$

CREATE PROCEDURE sp_obtener_siguiente_numero(
    IN p_codigo_empresa VARCHAR(20),
    IN p_tipo_documento ENUM('presupuesto','factura','factura_proforma','abono'),
    OUT p_numero_completo VARCHAR(50)
)
BEGIN
    DECLARE v_serie VARCHAR(10);
    DECLARE v_numero_actual INT;
    DECLARE v_anio VARCHAR(4);
    
    SET v_anio = YEAR(CURDATE());
    
    IF p_tipo_documento = 'presupuesto' THEN
        SELECT serie_presupuesto_empresa,
               numero_actual_presupuesto_empresa + 1
        INTO v_serie, v_numero_actual
        FROM empresa
        WHERE codigo_empresa = p_codigo_empresa AND activo_empresa = TRUE;
        
        SET p_numero_completo = CONCAT(v_serie, v_anio, '-', LPAD(v_numero_actual, 4, '0'));
        
    ELSEIF p_tipo_documento = 'factura' THEN
        SELECT serie_factura_empresa,
               numero_actual_factura_empresa + 1
        INTO v_serie, v_numero_actual
        FROM empresa
        WHERE codigo_empresa = p_codigo_empresa AND activo_empresa = TRUE;
        
        SET p_numero_completo = CONCAT(v_serie, v_anio, '/', LPAD(v_numero_actual, 4, '0'));
        
    ELSEIF p_tipo_documento = 'factura_proforma' THEN
        SELECT serie_factura_proforma_empresa,
               numero_actual_factura_proforma_empresa + 1
        INTO v_serie, v_numero_actual
        FROM empresa
        WHERE codigo_empresa = p_codigo_empresa AND activo_empresa = TRUE;
        
        -- Formato: FP2024/0001
        SET p_numero_completo = CONCAT(v_serie, v_anio, '/', LPAD(v_numero_actual, 4, '0'));
        
    ELSEIF p_tipo_documento = 'abono' THEN
        SELECT serie_abono_empresa,
               numero_actual_abono_empresa + 1
        INTO v_serie, v_numero_actual
        FROM empresa
        WHERE codigo_empresa = p_codigo_empresa AND activo_empresa = TRUE;
        
        -- Formato: R2024/0001
        SET p_numero_completo = CONCAT(v_serie, v_anio, '/', LPAD(v_numero_actual, 4, '0'));
    END IF;
END$$
```

```sql
CREATE OR REPLACE VIEW v_documentos_presupuesto AS
SELECT 
    -- Documento actual
    dp.id_documento_ppto,
    dp.id_presupuesto,
    dp.id_version_presupuesto,
    dp.tipo_documento_ppto,
    dp.numero_documento_ppto,
    dp.serie_documento_ppto,
    dp.subtotal_documento_ppto,
    dp.total_iva_documento_ppto,
    dp.total_documento_ppto,
    dp.fecha_emision_documento_ppto,
    dp.ruta_pdf_documento_ppto,
    dp.motivo_abono_documento_ppto,
    dp.observaciones_documento_ppto,
    
    -- Presupuesto
    p.numero_presupuesto,
    p.nombre_evento_presupuesto,
    
    -- Cliente
    c.nombre_cliente,
    c.apellido_cliente,
    CONCAT(c.nombre_cliente, ' ', c.apellido_cliente) AS nombre_completo_cliente,
    
    -- Empresa emisora
    e.nombre_empresa,
    e.nif_empresa,
    
    -- Documento origen (para abonos)
    do.id_documento_ppto AS id_documento_origen,
    do.numero_documento_ppto AS numero_documento_origen,
    do.tipo_documento_ppto AS tipo_documento_origen,
    do.total_documento_ppto AS total_documento_origen,
    
    -- Auditoría
    dp.created_at_documento_ppto,
    dp.updated_at_documento_ppto

FROM documento_presupuesto dp

INNER JOIN presupuesto p 
    ON dp.id_presupuesto = p.id_presupuesto

INNER JOIN cliente c 
    ON p.id_cliente = c.id_cliente

INNER JOIN empresa e 
    ON dp.id_empresa = e.id_empresa

LEFT JOIN documento_presupuesto do 
    ON dp.id_documento_origen = do.id_documento_ppto

WHERE dp.activo_documento_ppto = 1

ORDER BY dp.fecha_emision_documento_ppto DESC, dp.id_documento_ppto DESC;
```

### 5.1 Flujo: Generar Parte de Trabajo

### 5.2 Flujo: Generar Factura Proforma

1. Usuario presiona [Factura Proforma]
2. Valida: presupuesto aprobado + no existe factura proforma
3. Genera número con `sp_obtener_siguiente_numero('factura_proforma')`
4. Crea PDF con literal "FACTURA PROFORMA"
5. Registra en BD y actualiza contador
6. Descarga PDF

### 5.3 Flujo: Registrar Pago con Factura ⭐ ACTUALIZADO

1. Usuario abre modal [Registrar Pago]
2. Sistema verifica si ya existe alguna factura emitida:
   - **SI existe**: Obtiene `id_empresa` de primera factura (empresa bloqueada)
   - **NO existe**: Permitirá seleccionar empresa
3. Completa: tipo, importe, fecha, método
4. Si marca checkbox "Generar factura":
   - **4A. Sistema muestra dropdown de selección de empresa real**
   - **SI empresa bloqueada**: Dropdown aparece DESHABILITADO con empresa ya seleccionada
   - **SI NO bloqueada**: Carga empresas mediante `sp_listar_empresas_facturacion()` (solo empresas NO ficticias)
   - Usuario **DEBE** seleccionar empresa emisora (o usar la bloqueada)
   - **Validación**: NO permitir continuar si no hay empresa
   - **Validación**: Rechazar si empresa seleccionada es ficticia (doble check backend)
   - **Validación**: Si existe factura previa, validar que `id_empresa` coincida
   - Tipo='total' → Factura proforma con empresa seleccionada
   - Tipo='anticipo' → **El usuario elige el tipo de documento**:
     - Opción A: **Factura de anticipo** (factura normal, abonable si hay error)
     - Opción B: **Factura proforma de anticipo** (no abonable — en caso de error se repite con nueva numeración)
   - En ambas opciones se registra el importe pagado y se vincula el documento
5. Registra documento con `seleccion_manual_empresa_documento_ppto = TRUE`
6. Registra pago y vincula documento
7. Actualiza resumen de pagos

**Notas importantes**: 
- El presupuesto original usa empresa ficticia, pero **TODA factura (incluso proforma) SIEMPRE usa empresa REAL** seleccionada manualmente para tener validez fiscal
- **Una vez emitida la primera factura, la empresa queda BLOQUEADA** para todas las facturas futuras de ese presupuesto
- Los abonos heredan automáticamente la empresa de la factura que abonan

### Decisiones técnicas clave sobre ABONOS

1. **Solo facturas normales**: NO se abonan facturas proforma (ni las de pago 100% ni las de anticipo)
2. **Proforma de anticipo = se repite, no se abona**: Si el cliente eligió factura proforma en un anticipo y hay un error, la solución es emitir una nueva proforma (marcar la anterior como `activo=0`) — **no se genera rectificativa**
3. **Factura real de anticipo = sí se abona**: Si el cliente eligió factura de anticipo (real) y hay error, se aplica el flujo normal de abono/rectificativa
4. **Un abono por factura**: Una factura normal = UN abono (no múltiples en v1)
5. **Anulación automática**: Al generar abono, pago se anula automáticamente
6. **Importes negativos**: Abonos SIEMPRE negativos (validación estricta)
7. **Motivo obligatorio**: Aparece en PDF
8. **Doble confirmación**: Checkbox + SweetAlert
9. **Auditoría completa**: RegistroActividad

### Decisiones técnicas clave sobre EMPRESAS REALES

1. **Presupuestos con empresa ficticia**: Presupuestos y partes trabajo usan empresa ficticia principal
2. **TODAS las facturas con empresa real**: Factura proforma, anticipo y normal SIEMPRE requieren empresa REAL (no ficticia)
3. **Selección manual obligatoria**: Usuario DEBE seleccionar empresa real al emitir primera factura de un presupuesto
4. **Empresa bloqueada tras primera factura**: Una vez emitida cualquier factura (proforma, anticipo, normal), la empresa queda BLOQUEADA para ese presupuesto
5. **No cambiar empresa**: NO se permite cambiar de empresa tras emitir factura (validación frontend disabled + backend rechaza)
6. **Abonos heredan empresa**: Los abonos usan automáticamente la `id_empresa` de la factura que abonan
7. **Validación doble**: Frontend deshabilita campo si empresa bloqueada + Backend valida coincidencia con primera factura
8. **Query empresa bloqueada**: `SELECT id_empresa FROM documento_presupuesto WHERE id_presupuesto = ? AND tipo_documento_ppto IN ('factura_proforma','factura_anticipo','factura_normal') LIMIT 1`
